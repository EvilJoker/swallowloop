# SwallowLoop 宪法

## 核心原则

### I. 人类决策权（不可协商）

每个流水线阶段都需要人类审批才能进入下一阶段。AI 产出 Markdown 文档，人类保留最终决策权。任何阶段的打回必须包含可操作的反馈意见。

**理由**：防止 AI 失控执行；确保质量与用户意图一致。

### II. 流水线透明性

所有 AI 产出必须存储为 Markdown 文档，并标明来源。阶段转换、批准和打回必须记录时间戳和评论。

**理由**：实现从 Issue 创建到 PR 提交的全链路可审计性和可追溯性。

### III. 测试优先原则

每次功能修改必须包含对应测试。提交前所有测试必须通过。新阶段契约和服务间通信必须有集成测试。

**理由**：防止回归；测试是预期行为的活文档。

### IV. DDD 架构合规性

依赖方向规则（强制执行）：
- Domain → Application → Infrastructure → Interfaces
- Domain 层禁止依赖任何外层
- Infrastructure 层只能依赖 Domain 接口（ABC + @abstractmethod）
- 禁止跨层反向依赖

**理由**：确保可维护性和可测试性；防止紧密耦合。

### V. 最小化接口原则

每个模块只暴露必要的方法（通常 3-5 个）。内部实现细节使用 `_` 前缀。使用 `__all__` 显式声明公开 API。

**理由**：约束 AI 行为；防止随意访问内部状态，促进清晰的边界。

## 开发工作流

### 代码审查要求

- 所有变更必须经过审查才能合并
- 审查必须验证：架构合规性、测试覆盖率、无硬编码密钥
- 提交信息格式：`type(scope): description`（ Conventional Commits 规范）

### 质量门禁

1. **本地**：`pytest tests/ -v` 必须通过
2. **E2E**：修改前端/后端 API 时，需按 CLAUDE.md 运行完整 E2E 验证
3. **架构**：提交 PR 前运行契约检查

### 测试策略

| 变更类型 | 必需测试 |
|---------|---------|
| 新功能 | 单元测试 + 集成测试 |
| Bug 修复 | 回归测试 |
| API 变更 | E2E 验证 |
| 阶段/任务逻辑 | Handler 单元测试 |

## 治理

**宪法版本**: 1.0.0 | **批准日期**: 2026-04-07 | **最后修订**: 2026-04-07

### 修订程序

1. 通过带 `[constitution]` 标签的 Issue 提出变更
2. 记录修订理由和影响
3. 需要项目维护者批准
4. 版本号递增遵循语义化版本：
   - MAJOR：不兼容的原则移除或重定义
   - MINOR：新增原则或大幅扩展
   - PATCH：澄清、措辞、笔误修复

### 合规性

- 所有 PR 必须自检宪法合规性
- 架构契约检查通过 `.claude/skills/architecture-design` 自动运行
- API 变更需要 E2E 验证（按 CLAUDE.md 第：前后端代码改动验证规范）
