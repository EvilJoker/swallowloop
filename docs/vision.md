# SwallowLoop · 燕子回环：项目设想

> **一句话概念**  
> **SwallowLoop · 燕子回环：围绕你的代码仓，一圈一圈像搭建巢穴一样，维护你的项目。**

## 一、项目愿景

- 为**程序员个人 / 小团队**提供一个**常驻的开发 Agent 系统**：  
  只需配置仓库地址和 API Key，就能在不改变现有 Git 工作流的前提下，模拟「有一个远程实习生帮你写代码」。
- 重点服务 **单代码仓库、中大型代码库** 的日常维护与演进，包括：
  - 小功能实现
  - Bug 修复
  - 日常重构与技术债治理
  - 测试与文档补齐

## 二、核心设计思想

1. **围绕仓库打转，而不是取代开发者**
   - 一切改动都以 **PR（Pull Request）** 的形式交付，人类保留最终审批权。
   - Agent 像每年回巢的燕子，围着同一个仓库「一圈一圈」巡检、搭建、修补。

2. **最小侵入：兼容现有 Git / GitHub 流程**
   - 不强行接管现有 CI / 分支策略，而是在其之上增加一层「自动劳动」：
     - 从 Issue / 任务出发
     - 在 fork / 分支上开发
     - 开 PR 等待人工审查

3. **安全优先：隔离 + 审计**
   - 所有自动改动只发生在：
     - Agent 专用 fork 仓库，或
     - Agent 专用工作目录 / 分支
   - 主仓只接受 PR，且必须通过 CI 和人工 Review。

4. **闭环与进化**
   - 每个任务从「规划 → 开发 → 测试 → PR → 审批」形成可追踪闭环。
   - 从失败 PR / CI 错误 / Review 意见中抽取经验，逐步沉淀到「项目记忆」，让 Agent 越来越懂某个仓库。

## 三、目标使用体验

对用户（开发者 / 仓库维护者）来说，理想体验是：

1. 在 Web 前端或 GitHub Issue 中创建 / 标记任务（例如打上 `swallow` 标签）。
2. SwallowLoop 自动：
   - 读取任务与相关代码
   - 生成简短设计说明
   - 在 fork / 分支上实现代码修改并跑测试
   - 提交 PR，并在前端面板聚合「待我审批的 PR」列表
3. 你只需要：
   - 定期打开前端页面，查看「待审批」任务与对应 PR
   - 在 GitHub 上做正常的 Review / Merge / 打回

简单说：**「人类做决策与审查，燕子帮你干重复活。」**

## 五、后续演进方向

- **并行任务调度**
  - 支持多个任务同时进行，控制最大并发数

- **巡检与技术债治理**
  - 定期扫描仓库（TODO/FIXME、重复代码、复杂度等），自动生成建议任务

- **经验 / 风格记忆**
  - 将被打回的 PR / Review 意见结构化存储，作为后续任务的额外上下文

- **支持非 GitHub 场景**
  - GitLab / Gitea / 本地 Git 仓库等

- **Web 前端面板**
  - 任务列表和状态展示
  - 待审批 PR 聚合视图
