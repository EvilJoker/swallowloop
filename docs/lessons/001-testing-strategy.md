# 测试策略与 API 格式一致性

## 问题描述

测试环境与实际运行环境不一致，导致测试通过但实际使用失败。

## 典型案例

### API 格式不一致

**问题**：
- 后端返回 `{"issue": {...}}` 但前端期望直接返回 `Issue` 对象
- 导致 E2E 测试看到 "Unknown error" 但测试未捕获

**教训**：
- 前后端 API 响应格式必须一致
- 发现 E2E 测试中看到 "Unknown error" 时应立即怀疑是 API 格式不匹配
- 不应该依赖网络响应检查，而应该验证 API 格式

### 检查清单

- [ ] 后端每个 API 端点返回什么格式？
- [ ] 前端 `handleResponse` 如何解析响应？
- [ ] 是否有端点返回 `{"issue": ...}` 包装格式？
- [ ] 前端组件测试使用 mock API 响应时是否与真实 API 格式一致？

### 测试环境 vs 实际环境

**问题**：
- E2E 测试依赖真实后端服务，但测试环境没有启动后端
- 前端组件测试使用 mock 数据，与真实 API 格式脱节

**教训**：
- E2E 测试应该在完整环境（后端+前端）中运行
- 或者 mock 网络请求确保测试可靠性
- Playwright 的 `waitForResponse` 可以帮助验证 API 调用是否成功

## 根因分析框架

每次发现 bug 后，问自己：

1. **为什么测试没发现？**
   - 测试用例覆盖不足？
   - 测试假设与实际不符？
   - 测试环境和运行环境不同？

2. **如何修复测试？**
   - 添加新的测试用例？
   - 修改现有测试用例的假设？
   - 统一测试环境和运行环境？

3. **如何防止类似问题？**
   - 添加到检查清单？
   - 添加到 CI 流程？
   - 添加自动化检测？
